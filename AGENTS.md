### 1. Scraping-Layer (Apify Provider-Chain & yt-dlp Fallback)

* **Technologie:** Apify API Client (`apify-client`).
* **Pluggable Provider-Chain (`backend/src/scrapers/providers/`):** Das Social-Media-Scraping ist als erweiterbare Provider-Kette implementiert. Jeder Provider kapselt einen Scraper ΟΦber das `SocialScrapeProvider`-Interface (`name`, `scrape()`) und liefert ein normalisiertes `ScrapingResult` (`caption`, `videoUrl`, `imageUrl`, `authorHandle`). * **Neuen Fallback-Actor hinzufΟΦgen:** Einen Provider unter `backend/src/scrapers/providers/` implementieren (Actor-spezifisches `scrape()`) und dem Array in `backend/src/scrapers/providers/index.ts` hinzufΟΦgen βÄî kein Eingriff in die Aufruflogik oder `.env` nΟΕtig.
* **Aktiver Provider (eigener First-Party Actor):** **Social Video Downloader** (`<username>/social-video-downloader`, Quellcode in einem separaten Schwester-Repo `../apify-actor` neben diesem Projekt). Unser eigener Apify-Actor umschlieΟüt `yt-dlp` hinter Apify **Residential-Proxies** und liefert `caption` (Post-Beschreibung), `imageUrl` (Thumbnail) und `authorHandle` sowie eine gemergte MP4-Datei, die im Apify **Key-Value-Store** abgelegt und als ΟΕffentlich abrufbare `videoUrl` (Plain-GET, ohne Token/Header) zurΟΦckgegeben wird. Damit sind Caption **und** Cover-Bild wiederhergestellt (die der vorherige Drittanbieter-Actor `rover-omniscraper/media-downloader-actor` nicht lieferte). Die Actor-ID wird per `APIFY_SOCIAL_ACTOR_ID` gesetzt (nach `apify push`); ohne gesetzte ID fΟΛllt die Kette auf den lokalen yt-dlp-Fallback zurΟΦck. **Cookie-frei:** ZuverlΟΛssigkeit kommt aus Residential-Proxies + Session-Rotation (bis zu 5 Versuche), nicht aus manuellen Cookies. Siehe `docs/apify.social-downloader.md` (Contract) und das README im separaten `apify-actor`-Repo.
* **Local Fallback fΟΦr YouTube/TikTok/Facebook:** Falls die gesamte Apify-Provider-Chain fehlschlΟΛgt oder blockiert wird, greift das System fΟΦr YouTube Shorts, TikTok und Facebook auf das lokale Command-Line Tool `yt-dlp` als robusten Fallback zurΟΦck.
* **Ergebnis:** Die Scraper-Pipeline liefert ein standardisiertes `ScrapingResult`-Objekt mit einer Caption, der Bild-Cover-URL (`imageUrl`) und einem direkt abspielbaren Medien-Link (`audioUrl` / `videoUrl`) fΟΦr die Transkription und Frame-Extraktion.
* **Bilderkarussell-Posts (Image Carousels):** Neben reinen Video-Reels unterstΟΦtzt die Extraction Queue auch Instagram-/TikTok-Posts, die aus einer Bilderreihe bestehen (RapidAPI `type: "multiple"` mit `medias[].type: "image"` βÄî eine hΟΛufige Form von Rezept-Posts, bei der Zutaten und Schritte als Text auf den Slides stehen). Der RapidAPI-Provider liefert dafΟΦr die neue `MediaDownload`-Variante `{ kind: 'images', imageUrls, headers }` (Slides in Post-Reihenfolge, gedeckelt auf 15), sobald ein Post **kein** Video enthΟΛlt. `downloadMedia()` lΟΛdt jede Slide fail-soft herunter (`imageFilePaths` in `DownloadedMedia`, Bytes flieΟüen in die Media-Metriken). Im Worker (`queue.ts`) ΟΦbernehmen die Slides die Rolle der Video-Frames: Aus ihnen wird das Kachel-Grid fΟΦr die bestehende Gemini-Best-Shot-Auswahl gebaut (Single-Slide-Posts ΟΦberspringen das Grid βÄî ffmpeg `xstack` braucht βâΞ 2 Inputs βÄî und laden die Slide direkt als Cover hoch; dieselbe Direkt-Upload-Logik dient als Fallback, wenn die Gemini-Auswahl scheitert). An `extractRecipe()` gehen die Slides **einzeln in voller AuflΟΕsung** (neuer Parameter `carouselImagePaths`) statt des herunterskalierten Grids, damit der auf den Bildern stehende Rezepttext OCR-lesbar bleibt; der Prompt enthΟΛlt eine eigene Carousel-Kontextklausel. Audio/Transkription entfΟΛllt, die Video-LΟΛngenprΟΦfung greift nicht (`duration: 0`). Cover-Bilder laufen ΟΦber denselben transienten Frame-Hand-off wie Video-Frames (kein serverseitiges Rehosting).
* **Benutzername/Handle-Extraktion & Link-Fallback:** Um fehlerhafte, unbrauchbare Creator-Profillinks zu vermeiden (wenn Scraper nur den Anzeigenamen wie "Ashley Markle" statt des technischen Usernames zurΟΦckgeben), extrahiert das Backend bevorzugt den technischen Usernamen aus dem Feld `owner.username` des RapidAPI-Ergebnisses (fΟΛllt bei Fehlen auf `author` zurΟΦck). Im Frontend validiert die Profil-Link-Generierung den extrahierten Handle ΟΦber eine Regex (keine Leerzeichen, etc.). Falls der Handle ungΟΦltig bzw. ein reiner Anzeigename ist, wird statt eines defekten Profil-Links direkt die `reelUrl` (das Originalvideo) verlinkt, wo der Benutzer den Creator leicht finden kann.

### 3. KI-Layer (Google Gemini)

* **Technologie:** `@google/generative-ai` SDK (Gemini 1.5/2.5/3.1/3.5 Flash).
* **Funktion:** Die Audiodatei wird ΟΦber die Google AI File API hochgeladen. Gemini verarbeitet Audio und Text (`caption`) in einem einzigen multimodalen Aufruf.
* **Structured Outputs & Clean Parsing:** Gemini wird durch ein strenges JSON-Schema gezwungen, das Rezept exakt nach einem detaillierten Schema (Titel, Beschreibung, Zutaten mit Mengen/Einheiten, Schritte, AusrΟΦstung, NΟΛhrwertschΟΛtzungen, Kochtipps, Alternativzutaten und ein passendes Rezept-Emoji) zu strukturieren.

  * **Rezept-Emoji-Generierung:** Gemini wΟΛhlt basierend auf dem Rezepttitel und der Hauptzutat ein passendes Emoji (z. B. πüΞî bei Kartoffelrezepten, πüçï bei Pizza), welches als visueller Platzhalter bei fehlenden Bildern dient.
  * **Bereinigung der Zutatennamen:** Zutatennamen (`name`) werden im Prompt explizit von Mengen, Zahlen, MaΟüeinheiten und Modifizierern/Eigenschaften (wie "leicht", "mager", "gerieben") gesΟΛubert; diese Daten flieΟüen sauber in die dedizierten Felder `amount`, `unit` und das neue `modifier`-Feld.
  * **NΟΛhrwerte pro Zutat:** Das Schema erzwingt fΟΦr jede Zutat NΟΛhrwertangaben (`calories`, `protein`, `carbs`, `fat`) bezogen auf die konkrete Zutat und die gesamte angegebene Menge (nicht pro 100g oder pro EinzelstΟΦck). Falls nicht ermittelbar, wird standardmΟΛΟüig 0 ausgegeben.
  * **Dekomposition von Verbundzutaten:** Im Prompt ist geregelt, dass wΟΛhrend des Rezept-Videos zubereitete Verbundkomponenten (wie "Smash Burger Patties" oder "selbstgemachtes Pesto") in ihre atomaren Rohbestandteile zersetzt werden mΟΦssen (z.   * **Mehrfachrezept-Erkennung (AmbiguitΟΛts-Check):** Videos und Bilderkarussells kΟΕnnen mehrere eigenstΟΛndige Rezepte enthalten (z. ## πü©Κ Health Check & Mobile Benachrichtigungs-Dienst

Das Projekt verfΟΦgt ΟΦber ein eigenstΟΛndiges Monitoring- und Alarmsystem im Ordner `healthcheck/`, das alle Kernkomponenten (Express Backend, Frontend-Website, RapidAPI-Scraper, Apify-Actor, Gemini-LLM-API) unabhΟΛngig ΟΦberwacht und den Administrator bei AusfΟΛllen oder Wiederherstellungen per Push-Benachrichtigung auf dem Smartphone (via **ntfy.sh** und/oder **Telegram Bot**) benachrichtigt.

### 1. Ο€berwachte Dienste & Test-Logik (`healthcheck/src/index.ts`)
* **Backend:** Ruft die ΟΕffentliche `/health` API auf und validiert das Antwort-JSON, welches die Uptime und den Verbindungsstatus der Datenbank zurΟΦckgibt.
* **Website:** FΟΦhrt einen HTTP GET-Request gegen die in `HEALTHCHECK_WEBSITE_URL` hinterlegte URL aus und prΟΦft auf HTTP-Status `200-299`.
* **RapidAPI:** Sendet einen ressourcenschonenden Test-POST an `https://<RAPIDAPI_HOST>/v1/social/autolink` mit einer Dummy-URL, um API-Key-GΟΦltigkeit und Routing ohne Scraping-Kosten zu prΟΦfen. HTTP `401`/`403` deutet auf einen ungΟΦltigen API-Key hin, `5xx` auf einen API-Ausfall.

### 2. Persistente Zustandshaltung & Entprellung

* Beim ersten Start des Dienstes wird ein initialer Alarm gesendet, falls ein Dienst fehlerhaft (`down`) startet.

* **Telegram Bot:** Wenn `TELEGRAM_BOT_TOKEN` und `TELEGRAM_CHAT_ID` gesetzt sind, sendet das Backend HTML-formatierte Telegram-Nachrichten direkt auf das Smartphone des Administrators.

### 4. AusfΟΦhrung & Integration
